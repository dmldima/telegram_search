# План аудита багов и неэффективностей

Проект: `telegram_search` — мониторинг Telegram-каналов по ключевым словам/regex
с пересылкой совпадений в личку через бота. Запускается раз в сутки через GitHub
Actions (`.github/workflows/monitor.yml`), состояние хранится в
`processed_messages.json` и коммитится обратно в репозиторий.

Дата составления: 2026-06-30.

---

## 1. Цель и объём аудита

**Цель:** найти и приоритизировать ошибки корректности (баги) и неэффективности
(лишние API-вызовы, рост репозитория, хрупкие места), составить backlog
исправлений с критериями приёмки.

**Объём (что смотрим):**
- `telegram_monitor.py` — вся логика (~466 строк).
- `.github/workflows/monitor.yml` — окружение запуска, конкурентность, git push.
- `requirements.txt` — версии зависимостей.
- `processed_messages.json` — формат и стоимость хранения состояния в git.

**Вне объёма:** функциональность Telethon/Telegram API сама по себе; настройка
секретов GitHub; генерация session string.

---

## 2. Методология

Аудит в 5 проходов, каждый с конкретным результатом:

1. **Статический разбор потока данных** — проследить путь сообщения от
   `iter_messages` до отправки; найти места, где состояние меняется раньше, чем
   подтверждён результат.
2. **Анализ обработки ошибок** — каждый `try/except`: что глотается, что
   теряется, что может зациклить/заспамить.
3. **Анализ времени и часовых поясов** — все `datetime.now()` / `.replace(tzinfo=None)`
   / сравнения дат.
4. **Анализ окружения запуска** — поведение в GitHub Actions (эфемерный раннер,
   UTC, конкурентные запуски, git push).
5. **Ревью эффективности** — лишние сетевые вызовы, рост `processed_messages.json`
   и истории git, устаревшие версии actions/зависимостей.

**Инструменты для прогона:**
- `python -m py_compile telegram_monitor.py` — синтаксис.
- `ruff check` / `pyflakes` — мёртвый код, неиспользуемые импорты.
- `bandit` — небезопасные `except: pass`, eval-подобные места.
- Ручной запуск с фиктивными env-переменными для проверки путей валидации.

## 3. Предварительный backlog находок

Найдено при первичном чтении кода. Приоритеты: **P0** — потеря данных/падение,
**P1** — заметное некорректное поведение, **P2** — эффективность/надёжность,
**P3** — косметика.

### P0 — критичные

| # | Файл:строка | Проблема | Последствие |
|---|---|---|---|
| 1 | `telegram_monitor.py:214` | Сообщение помечается обработанным (`processed_dict[unique_id] = ...`) **до** успешной пересылки. Если все попытки отправки падают, сообщение уже «обработано». | Совпавшее сообщение теряется навсегда при временном сбое отправки. |
| 2 | `telegram_monitor.py:443-446` | При исключении в цикле наборов `save_processed_messages` (строка 433) не вызывается — он внутри `try`, а `except` идёт мимо сохранения. | Все отметки «обработано» за прогон теряются → на следующем запуске дубли пересылаются повторно. |
| 3 | `telegram_monitor.py:12,16` | `int(os.getenv('TELEGRAM_API_ID'))` и `int(os.getenv('YOUR_USER_ID'))` выполняются на уровне модуля. Если переменная отсутствует → `int(None)` → `TypeError` при импорте. | Скрипт падает с непонятным трейсбеком **до** дружелюбной проверки `missing` в `main()` (строки 351-355) — эта проверка для данных переменных мёртвая. |

### P1 — функциональные

| # | Файл:строка | Проблема | Последствие |
|---|---|---|---|
| 4 | `telegram_monitor.py:194,199,214` | Время: `datetime.now()` — локальное, `message.date` — UTC (приводится через `.replace(tzinfo=None)`). | В UTC-раннере GitHub Actions работает «случайно». В любом другом часовом поясе окно `TIME_RANGE_HOURS` смещается → пропуск или повторный захват сообщений. Использовать `datetime.now(timezone.utc)`. |
| 5 | `telegram_monitor.py:250,264` | В `fallback_text` используется markdown (`**...**`), но `parse_mode` у бота не задан (по умолчанию в Telethon `None`). | Пользователю приходит литеральный текст `**[set_name] Пост из ...**` вместо жирного. |
| 6 | `telegram_monitor.py:223-238` | `bot.forward_messages(..., from_peer=channel)`, где `channel` — entity, разрешённый **пользовательским** клиентом. Бот, как правило, не имеет доступа к этому entity. | Первая попытка пересылки почти всегда падает (молча, `except: pass`) → лишний API-вызов на каждое совпадение, всегда уходим в fallback. |
| 7 | `telegram_monitor.py:153-156` | Исключающие слова ищутся как подстрока (`exclude_word in text_lower`) без границ слова. | Ложные исключения: `exclude=["акт"]` отсечёт «контакты», «характеристика» и т.п. Та же проблема с keywords (строка 164) — задокументирована как фича, но стоит подтвердить намерение. |

### P2 — надёжность и эффективность

| # | Файл:строка | Проблема | Последствие |

| 12 | `telegram_monitor.py:282-287` | Любая ошибка канала шлёт сообщение пользователю. | При недоступности нескольких каналов — спам уведомлениями за один прогон. |

## 4. План работ (порядок исправления)

1. **Спринт 1 — остановить потерю данных (P0):**
   находки #1, #2, #3. Помечать сообщение обработанным только после успешной
   отправки (или вести отдельный набор «к ретраю»); сохранять состояние в
   `finally`; перенести парсинг env в `main()` после проверки `missing`.
2. **Спринт 2 — корректность (P1):** #4 (UTC), #5 (parse_mode), #6 (убрать
   бессмысленный forward или проверять доступ), #7 (границы слов / подтвердить
   намерение).
3. **Спринт 3 — окружение и эффективность (P2):** #8 (`concurrency:` в workflow),
   #9 (pull --rebase + ретраи push), #10 (вынести состояние из git), #11, #12.
4. **Спринт 4 — гигиена (P3):** #13–#16.

---

## 5. Критерии приёмки

- **P0 закрыты:** при искусственном сбое отправки сообщение **не** теряется и
  повторно обрабатывается на следующем прогоне; прерывание в середине цикла не
  приводит к повторной массовой пересылке; запуск без обязательных env даёт
  понятное сообщение, а не трейсбек на импорте.
- **P1 закрыты:** временное окно корректно при не-UTC TZ (покрыто тестом);
  жирный текст отображается; нет «всегда падающего» первого forward.
- **Регрессий нет:** `python -m py_compile` проходит; ручной прогон с тестовым
  набором не шлёт дубли.
- **Эффективность:** один прогон делает ≤1 сетевой вызов отправки на совпадение;
  размер/рост `processed_messages.json` под контролем.

## 6. Статус исправлений (2026-07-01)

Исправлено в этой ветке (`telegram_monitor.py`, `.github/workflows/monitor.yml`):

- **#1 (P0)** — сообщение помечается обработанным только после успешной отправки;
  при сбое — ретрай на следующем прогоне. Отправка вынесена в `send_match()`.
- **#2 (P0)** — `save_processed_messages()` перенесён в `finally`: состояние
  сохраняется даже при исключении в середине прогона.
- **#3 (P0)** — парсинг `int(TELEGRAM_API_ID/YOUR_USER_ID)` перенесён в `main()`
  после проверки наличия переменных; при отсутствии/нечисловом значении —
  понятное сообщение вместо трейсбека на импорте.
- **#4 (P1)** — всё время переведено на `datetime.now(timezone.utc)`; сравнение
  с `message.date` (aware UTC) без `.replace(tzinfo=None)`. Легаси-timestamps без
  таймзоны корректно трактуются как UTC при очистке.
- **#5 (P1)** — убран markdown (`**...**`); отправка обычным текстом, чтобы
  произвольный текст канала не ломал разметку.
- **#6 (P1)** — удалён бесполезный `bot.forward_messages` (всегда падал) —
  минус один API-вызов на каждое совпадение.
- **#8 (P2)** — в workflow добавлен `concurrency` (нет параллельных прогонов).
- **#9 (P2)** — `git push` с `pull --rebase --autostash` и ретраями с
  экспоненциальной паузой.
- **#12 (P2)** — ошибки каналов накапливаются и шлются одной сводкой вместо
  сообщения на каждый канал.
- **#7 (P1)** — слова-исключения теперь матчатся по началу слова
  (`matches_word_start`, граница `\b`): `акт` больше не отсекает «контакты», но
  продолжает ловить «акты»/«актуальный». Ключевые слова намеренно оставлены
  поиском по подстроке (нулевой риск пропустить пост).
- **#10 (P2)** — состояние `processed_messages.json` вынесено из git в кэш
  GitHub Actions (`actions/cache`, ключ `processed-messages-<run_id>` +
  `restore-keys`). Убраны ежедневные коммиты «Update processed messages», шаг
  commit/push и права `contents: write` → `read`. Закоммиченный файл остаётся
  как стартовый seed на первый прогон.
- **#13 (P3)** — actions обновлены: `checkout@v4`, `setup-python@v5`, `cache@v4`.
- **#15 (P3)** — голый `except:` при снятии блокировки заменён на `except Exception:`.

Отложено намеренно:

- **#11** — персист сессии бота бесполезен в эфемерном раннере (свежий раннер
  каждый прогон), поэтому не трогали.

Проверено локально (`/py_compile`, YAML-валидация и юнит-тесты логики со
стабом telethon): дружелюбный выход при отсутствии/нечисловых env; отсутствие
потери данных при сбое отправки; корректная очистка легаси/aware/битых
timestamps.

## 7. Риски аудита

- Нет автоматических тестов — проверка изменений ручная/через мок Telethon.
- Реальное поведение пересылки ботом зависит от прав бота и типа канала —
  часть находок (#6) требует подтверждения на боевом окружении.
- Изменение формата хранения состояния (#10) затрагивает рабочий пайплайн —
  внедрять с миграцией и обратной совместимостью (как уже сделано для
  list→dict в `load_processed_messages`).
